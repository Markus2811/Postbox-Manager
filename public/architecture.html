Postbox Manager – Architektur

Übersicht: wo Frontend und Backend laufen, wie Supabase Auth, Postgres mit RLS und Storage zusammenspielen, und wo OpenAI eingebunden ist. Diese Seite liegt im Projekt unter public/architecture.html – lokal z. B. mit npm run dev unter http://localhost:3000/architecture.html öffnen (oder die Datei direkt im Browser, wenn CDN erreichbar ist).

Auf Vercel (deine App)
  • Frontend: gerenderte HTML/React-Seiten, Client-Bundles (u. a. Dashboard, Upload-UI).
  • Backend: dieselbe Next.js-App – API-Routen und Server-Logik laufen als Serverless-Funktionen (Node), kein eigener dauerhafter Server.
  • Geheimnisse: OPENAI_API_KEY, optional SUPABASE_SERVICE_ROLE_KEY nur serverseitig; NEXT_PUBLIC_* im Browser.
Bei Supabase (separates Produkt)
  • Datenbank: Postgres mit RLS – jede Zeile an auth.uid() gebunden.
  • Auth: Login/Session; Cookies werden von Next Middleware und API mit Anon-Key validiert.
  • Storage: Dateien im Bucket documents; Policies erlauben nur den eigenen Ordner.

Frontend vs. Backend (klare Trennung)

  • Frontend (sichtbar im Browser): Navigation, Formulare, Tabellen, Upload-Fortschritt. Nutzt NEXT_PUBLIC_SUPABASE_URL und NEXT_PUBLIC_SUPABASE_ANON_KEY für direkten Storage-Upload nach signierter URL. Keine Service-Role, kein OpenAI-Key im Client.
  • Backend (nur Server): prüft Session (auth.getUser()), spricht mit Postgres und Storage mit derselben User-Session (RLS greift), ruft OpenAI für Analyse/Fragen auf, setzt Metadaten.
  • Datenhaltung: alle persistenten Daten liegen bei Supabase (DB + Storage), nicht auf der Vercel-Festplatte.

RLS kurz erklärt

  • Tabellen documents und document_metadata: Policies erlauben nur Zeilen mit eigener user_id für Lesen/Schreiben/Löschen.
  • Storage: Objektpfad beginnt mit user_id/…; Policy vergleicht ersten Ordner mit auth.uid().
  • Die App-Queries filtern oft zusätzlich mit .eq("user_id", …) – RLS bleibt die eigentliche Absicherung in der Cloud.
